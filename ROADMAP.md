## Этап 1 — Новый анализатор: STFT + мультибэнд Spectral Flux 🔴

**Цель:** заменить `energies[]` (RMS одного каналa) на честный STFT со спектральным флюксом по нескольким полосам частот. Это база для всего остального.

### 1.1 FFT (`audio/stft.js`) 🟡
- Реализовать **radix-2 iterative FFT** (in-place, Cooley-Tukey). Референс: Numerical Recipes / Rosetta Code, ~80 строк.
- Функции:
  - `fftInPlace(re: Float32Array, im: Float32Array)` — размер = степень двойки.
  - `hannWindow(N): Float32Array` — предвычисленное окно.
  - `frameSignal(signal, frameSize=2048, hop=512, window)` → генератор кадров.
  - `magnitudeSpectrum(re, im): Float32Array` — только положительные частоты (N/2+1).
- **Параметры:** frame=2048, hop=512 при sr=44100 → ~86 кадров/сек, разрешение по частоте ~21.5 Hz.
- Юнит-тесты: sin(440Hz) → пик в bin 20 (при sr=44100, N=2048).

### 1.2 Мультибэнд Spectral Flux (`audio/spectralFlux.js`) 🟡
- Разделить спектр на **6 полос**: sub-bass 20-60, bass 60-250, low-mid 250-500, mid 500-2k, high-mid 2k-4k, high 4k-16k (Hz).
- Для каждого кадра t и каждой полосы b:
  ```
  flux[b][t] = Σ max(0, |X[t,k]| - |X[t-1,k]|)   для k в полосе b
  ```
- **Half-wave rectified**, log-compression: `log(1 + C*flux)` (C=1000) — снижает динамический разброс.
- Дополнительно: **complex-domain onset detection function** для этапа 1.3 (использует фазу для точнее вокала/сустейнов).
- Экспорт: `fluxBands: Float32Array[6]` + `fluxTotal: Float32Array`.

### 1.3 Адаптивные пороги + пиковый детектор (`audio/onsets.js`) 🟡
- Заменить фиксированный `mean + sens*std` на **local adaptive threshold**:
  ```
  threshold[t] = median(flux[t-W..t+W]) * α + δ
  ```
  где W ≈ 43 кадра (~0.5 сек), α=1.5, δ — небольшой floor.
- **Peak picking** (по Böck & Widmer 2013):
  - Локальный максимум в окне ±3 кадра.
  - Выше threshold[t].
  - Прошло ≥ `minInterval` от предыдущего пика в этой полосе (для percussive 60ms, harmonic 120ms).
- Возврат: per-band onset lists + confidence per onset.

### 1.4 Web Worker (`audio/worker.js`) 🟡
- Весь тяжёлый расчёт — в worker. Основной поток шлёт `ArrayBuffer` (Transferable) → получает готовый список онсетов.
- Прогресс-сообщения каждые ~100 мс → плавный прогресс-бар в UI.
- Fallback: если Worker недоступен (редко) — выполнить синхронно.

### 1.5 Интеграция
- `audio/analyzer.js` становится тонким фасадом:
  ```js
  async function analyzeTrack(audioBuffer, opts) {
    const monoPCM = downmix(audioBuffer);
    const result = await runInWorker({ pcm: monoPCM, sr, opts });
    return result; // { onsets, bands, bpm, holds, energies }
  }
  ```
- Старую `analyzeTrack` сохранить как `analyzeTrackLegacy` для A/B до финального удаления.

**Deliverable этапа:** плотность/качество онсетов заметно выше на любых треках; drum+vocal одновременно детектятся, а не только выбранный режим.

## Этап 2 — HPSS (Harmonic/Percussive Separation) 🔴

**Цель:** разделить спектрограмму на гармоническую (сустейны, вокал, гитара) и перкуссионную (кики, снейры, хэты) части. Ноты-тапы генерим из percussive, HOLD — из harmonic.

**Алгоритм — Fitzgerald 2010, median filtering:**

Пусть `S[t,k]` — magnitude spectrogram (N кадров × K бинов).

1. **Percussive enhance:** для каждого бина k применить медианный фильтр по оси времени, окно `Lh` (например, 17 кадров ≈ 200 мс).
   ```
   P_est[t,k] = median(S[t-Lh/2 .. t+Lh/2, k])
   ```
   (Perc-события — короткие всплески во времени; медиана по времени их подавляет → остаётся гармоническая часть. Нет, наоборот: медиана по времени = «то, что стабильно во времени» = harmonic. См. п. 2.)

2. **Harmonic enhance:** для каждого t применить медианный фильтр по оси частоты, окно `Lp` (17 бинов ≈ 350 Hz).
   ```
   H_est[t,k] = median_freq(S[t, k-Lp/2 .. k+Lp/2])
   ```
   (Harmonic — узкополосные линии; медиана по частоте их размазывает → остаётся перкуссия.)

Прошу прощения, перепутал. Правильно:
- **H_est** (harmonic estimate) = median **по времени** — гармонические компоненты стабильны во времени.
- **P_est** (percussive estimate) = median **по частоте** — перкуссия широкополосна в частоте.

3. **Soft mask** (Wiener-like):
   ```
   M_H = H_est² / (H_est² + P_est² + ε)
   M_P = P_est² / (H_est² + P_est² + ε)
   H = S · M_H
   P = S · M_P
   ```

4. Дальше — spectral flux считаем отдельно по H и P:
   - `flux_P` → **percussive onsets** (тапы, для DRUMS-режима).
   - `flux_H` → базис для HOLD-детекции (см. этап 3).

### Задачи (`audio/hpss.js`) 🔴
- [ ] Быстрая медиана скользящим окном (heap или сортированный skiplist). Для окна 17 — простой отсортированный массив с бинарной вставкой достаточен, O(N·L·log L).
- [ ] Оптимизация: обрабатывать только магнитуды (без фазы), фазу переиспользовать из оригинала при обратном STFT (если понадобится сам audio).
- [ ] Параметры: `Lh=17, Lp=17` (тюнить на реальных треках).
- [ ] Прогон в worker — HPSS занимает ~2-4× дольше STFT, критично для UX.
- [ ] Опция `hpssMode: 'off' | 'soft' | 'hard'` в настройках — если пользователь не хочет ждать.

**Deliverable:** DRUMS-режим работает даже на треке с гитарой, VOCAL-режим не срабатывает на кике.

## Этап 3 — Hold Notes v2 + структурный анализ 🔴

### 3.1 Детекция длинных нот (`audio/holds.js`) 🟡
- Работаем с **harmonic-спектрограммой** (из HPSS).
- Для каждого onset проверяем, есть ли устойчивая гармоническая энергия после него:
  ```
  sustainScore = mean(H_energy[t_onset .. t_onset + window]) / peak_at_onset
  ```
- Дополнительно: **pitch tracking** через autocorrelation или YIN (упрощ.) — если f0 стабильна в течение ≥300 мс с σ<3%, это гарантированный HOLD.
- Конец ноты: момент, когда `H_energy` падает ниже `0.35 * peak` или f0 сильно меняется.
- **Слияние близких онсетов** (< 80 мс) в один HOLD.

### 3.2 Определение начала/конца точно 🟢
- Уточнение начала — по complex-domain flux (пик может быть на 10-20 мс раньше энергетического).
- Конец — hysteresis: 2 порога, чтобы не «дёргать» на модуляциях.

### 3.3 Синкопы и акценты 🟡
- Определить сильные доли из BPM-сетки (см. этап 4).
- Онсеты не на сильных долях с высокой энергией → **синкопы** → пометить `accent:true`.
- Вокальные акценты: пики в 2-4 kHz на harmonic-части, короткие всплески RMS.

**Deliverable:** карты с осмысленным чередованием тапов и HOLD, длина HOLD соответствует реальным сустейнам в музыке.

## Этап 4 — BPM v2: устойчивый темп 🟡

Текущий `detectBPM` использует IOI-гистограмму — работает, но ломается на треках с сильной синкопой или переменным темпом.

### 4.1 Autocorrelation-based tempo (`audio/bpm.js`)
- Взять **onset novelty function** (`fluxTotal` из этапа 1) и посчитать её autocorrelation в диапазоне лагов 200-1000 мс (60-300 BPM).
- Пик автокорреляции → период удара.
- Комбинировать с существующей IOI-гистограммой (weighted vote).

### 4.2 Tempogram (опционально) 🟡
- Локальная autocorrelation по скользящим окнам (~8 сек) → 2D-темпограмма.
- Позволяет отследить смену темпа посреди трека.

### 4.3 Beat tracking (для сетки) 🟡
- Dynamic programming (Ellis 2007): максимизировать сумму (onset strength) - λ·(отклонение от периода).
- Результат: **beat times[]** — точная сетка ударов на весь трек.
- Используется для snap-to-grid HOLD-длин и синкоп.

### 4.4 Half/double tempo disambiguation
- Часто детектор путает 90 и 180 BPM. Оценка через:
  - Спектральный центроид ударов (перкуссия чаще на «правильном» темпе).
  - Пользовательский хинт `expectedTempoRange` в настройках.

**Deliverable:** BPM в HUD стабильно совпадает с реальным ±1 на 95% треков; сетка бит нарисована в редакторе.

## Этап 5 — Тайминги, калибровка, FPS-независимость 🟡

### 5.1 Калибровка задержки (`ui/calibration.js`) 🟡
- **Аудио-задержка** (audio latency): пользователь слышит метроном и тапает — вычисляем `audioOffset`.
- **Визуальная задержка** (video latency): нота появляется, пользователь тапает — вычисляем `visualOffset`.
- Итоговый оффсет = `audioOffset - visualOffset` (мс), сохраняем в `localStorage`.
- Отдельная страница «Калибровка» в меню, 10 тапов, показ σ (стабильность).

### 5.2 Компенсация в игре 🟢
- В `songTime()` вычитать `audioOffset`.
- В `render` время нот двигать на `visualOffset`.
- Референс: `audioCtx.baseLatency + audioCtx.outputLatency` (если доступно) как стартовая оценка.

### 5.3 FPS-независимость 🟢
- Уже частично есть (`dt` используется). Пройтись по всем местам, где движение зависит от FPS (частицы, bullets — там `dt` уже, ок; шейк камеры — сделать time-based, а не frame-based).
- Fixed-step для физики частиц при `dt > 33ms` (низкий FPS) — субшаги.

### 5.4 Более точные окна 🟢
- Текущие: PERFECT 95ms / GOOD 175ms / OK 245ms. Ужесточить:
  - MARVELOUS 25ms / PERFECT 45ms / GREAT 80ms / GOOD 140ms / OK 200ms / MISS.
- Опция сложности `judgeMode: 'lenient' | 'normal' | 'strict'` мультиплицирует окна.

**Deliverable:** после калибровки на любом железе PERFECT-рейт растёт на 20-40%.

## Этап 6 — Генератор карт v2 🟡

Текущий генератор случайно раскидывает онсеты по 4 дорожкам. Надо сделать **музыкальные** карты.

### 6.1 Паттерны (`game/mapgen.js`)
- Библиотека паттернов: **stream** (одиночные подряд), **jack** (одна дорожка N раз), **jumptrill** (перекрёстные), **chord** (2-4 одновременно), **stair** (0→1→2→3).
- Выбор паттерна по контексту: BPM, сложность, тип онсета (percussive/harmonic).

### 6.2 Ограничение плотности 🟢
- Максимум N нот в секунду по сложности: Easy=3, Normal=5, Hard=8, Expert=12+.
- Скользящее окно 1 сек, если превышено — дропаем слабейшие онсеты (по confidence).

### 6.3 Физическая играбельность 🟢
- Никаких 3+ нот одновременно (одну руку не расщепить).
- После chord — 100 мс тишины на этих дорожках.
- Jack (одна дорожка подряд) — минимум 120 мс интервал.

### 6.4 Заполнение пауз 🟢
- Если ≥ 3 сек без нот и в это время есть слабые онсеты (ниже threshold) — понизить threshold локально.
- Если ≥ 2 сек с плотностью > лимита — прорядить.

### 6.5 Автосложности 🟡
- Один анализ → 4 карты: Easy (только сильные онсеты + downbeats), Normal, Hard, Expert (все + акценты + chords).
- Выбор в меню перед стартом.

### 6.6 Учёт музыкальных фраз 🟡
- Простой sectioning по self-similarity matrix (уже есть спектрограмма) — определить границы куплетов/припевов.
- В припевах — плотнее и chord'ы, в куплетах — стрим.

**Deliverable:** карты ощущаются как «сыгранные человеком», а не «выплюнутые скриптом».

## Этап 8 — Постобработка: Bloom, Motion Blur, Trails 🟡

Canvas2D → пришлось бы через offscreen-canvas хаки. Лучше — **WebGL пост-слой поверх Canvas2D**.

### 8.1 Гибрид Canvas2D + WebGL (`render/postfx.js`) 🔴
- Основной game canvas остаётся 2D (проще для нот и UI).
- Второй canvas (WebGL) поверх: применяет к текстуре из 2D-canvas шейдеры:
  - **Bloom:** downsample → Gaussian blur → threshold → additive blend.
  - **Chromatic aberration:** сдвиг R/B каналов на ±1px к краям экрана.
  - **Vignette:** радиальное затемнение краёв.
  - **Motion blur:** накопительный буфер с fade 0.85 предыдущего кадра.
- Если WebGL не доступен — просто скрыть overlay, игра работает без постобработки.

### 8.2 Trails на нотах 🟢
- Нота оставляет затухающий след — 6 предыдущих позиций с уменьшающимся alpha.
- Delta для HOLD-нот — градиент вдоль tail'а, пульсирующий во время удержания.

### 8.3 Bloom только на ярких пикселях 🟡
- В шейдере: `if (luminance > 0.75) → bloom`. Настройка threshold в опциях.

**Deliverable:** визуал уровня Beat Saber / A Dance of Fire and Ice.

## Этап 9 — Music-reactive UI + фон 🟡

### 9.1 Live-анализ во время игры (`fx/musicReactive.js`) 🟢
- В игре — `AnalyserNode` (fftSize=1024) поверх playback source.
- Каждый кадр: `getByteFrequencyData` → массив 512 значений.

### 9.2 Пульсация фона 🟢
- Bass level (bins 0-10) → яркость радиального градиента фона.
- Beat-detect в реальном времени (простой RMS-flux) → короткая вспышка на битах.

### 9.3 Эквалайзер / спектрограмма 🟢
- 32 столбика внизу экрана, полупрозрачные, per-frame высота из frequency data.
- Опция в настройках: `bg: 'off' | 'pulse' | 'spectrum' | 'both'`.

### 9.4 Насыщенность цветов от энергии 🟢
- Global HSL saturation переменная растёт с RMS.

**Deliverable:** экран «дышит» вместе с музыкой, даже если не играешь — красиво.

## Этап 10 — Инфраструктура: кэш, редактор, реплеи 🟡

### 10.1 Кэш анализа (`audio/cache.js`) 🟢
- SHA-1 файла → ключ в IndexedDB.
- Хранить: `{ onsets, bpm, holds, beats, waveformPeaks }` (~50 КБ на песню).

### 10.2 Превью карты (`ui/mapPreview.js`) 🟢

### 10.3 Редактор карт (`ui/editor.js`) 🔴

### 10.4 Реплеи (`game/replay.js`) 🟡

### 10.5 Статистика 🟢

### 10.6 Экспорт/импорт карт 🟢
- JSON: `{ audio: filename, bpm, notes: [{time, endTime, lane, isHold}], meta: {...} }`.
- Совместимость: свой формат `.wtmap`, опционально импорт `.osu` (mania 4K).
